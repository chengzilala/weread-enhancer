# [OPEN] Debug Session: fast-switch-dark-text

## 1. 问题概述
- 现象：测试 `F1` 快速切换时，切到暗黑主题后正文文字看不见。
- 期望：即使在快速切换下，暗黑主题也应保持深灰背景 + 浅色清晰文字。
- 范围：`content.js` 中主题切换、计时器重刷、observer 重刷、跨主题样式清理链路。

## 2. 复现步骤
1. 打开微信读书阅读页
2. 快速执行：暗黑 → 护眼 → 明亮 → 暗黑
3. 每步间隔小于 1 秒
4. 观察最后暗黑状态下正文文字可见性

## 3. 初始假设
- H1：旧主题的定时器在最后一次暗黑切换后仍继续执行，覆盖了暗黑本应写入的浅色文字。
- H2：`clearLastPaintedThemeStyles()` 在快速切换中清掉了最后一次暗黑主题刚刷上的文字颜色。
- H3：最后一次暗黑时，真正可见的正文节点发生了重建，而新节点没有被及时刷成浅色。
- H4：快速切换时 `wreColorLastSet` 的 token 虽然递增了，但 observer 回调仍对旧主题节点生效。
- H5：暗黑模式下正文“看不见”并非文字是深色，而是某个可见容器背景/透明层与文字组合导致视觉不可见。

## 4. 当前状态
- 状态：已完成第一轮快速切换插桩，证据显示旧主题定时器不会覆盖最后一次暗黑结果。
- 当前阻塞：最新日志进一步表明，`readerChapterContent *` 的盲扫会优先命中 `readerControls` 与 `wr_tooltip_container`，仍未落到正文段落本身。
- 最新动作：在“正文文本节点取样”之外，新增 `.readerChapterContent` 前两层子树摘要输出，直接记录每层容器的 `tag/class/textLength/sampleText`，用于先锁定真正正文挂载在哪个子树上。
- 约束：在拿到正文页的新证据前，不修改主题业务逻辑。
